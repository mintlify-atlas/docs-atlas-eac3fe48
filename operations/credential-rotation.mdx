---
title: "Zero-downtime credential rotation procedures"
description: "Dual-credential window rotation for FSI Kafka API keys across Vault, Azure Key Vault, AWS Secrets Manager, and GCP Secret Manager."
---

Financial services regulations require a maximum 90-day credential lifetime for static API keys. The FSI Kafka Platform uses a **dual-credential window** pattern that keeps both old and new credentials active in Confluent Cloud during an overlap period, ensuring zero downtime while applications roll over to the new key. For applications using OAuth/OAUTHBEARER (recommended for Confluent Cloud per ADR-006), token rotation is fully automatic — no manual procedure is needed. This runbook covers rotation for **API keys** (required for the Confluent Terraform provider and supported as a fallback authentication mechanism) across the four supported secret backends.

<Note>
  OAuth/OAUTHBEARER tokens auto-rotate via the identity provider — no manual rotation is needed for application workloads using OAuth. This runbook applies only to API key rotation. See the [OAuth token rotation](#oauth-token-rotation) section for details.
</Note>

## Vault golden path

HashiCorp Vault KV v2 is the recommended credential backend for all Confluent Cloud scenarios (per D-04 and D-07). Each scenario directory includes a `vault.tf.example` reference pattern.

### Vault path convention

All secrets follow this prescriptive path (per D-06):

```
secret/fsi-kafka/{env}/{domain}/{sa-name}
```

| Segment | Example | Description |
|---|---|---|
| `{env}` | `prod`, `staging`, `dev` | Environment name |
| `{domain}` | `corebanking`, `fraud`, `compliance` | Business domain |
| `{sa-name}` | `account-txn-producer`, `fraud-alert-consumer` | Service account name |

Examples:
- `secret/fsi-kafka/prod/corebanking/account-txn-producer`
- `secret/fsi-kafka/staging/fraud/alert-signal-consumer`
- `secret/fsi-kafka/prod/compliance/screening-result-producer-sr`

### Vault Terraform reference pattern

```hcl
resource "vault_kv_secret_v2" "kafka_credentials" {
  mount = "secret"
  name  = "fsi-kafka/${var.environment}/${var.domain}/${var.sa_name}"

  data_json = jsonencode({
    api_key        = var.kafka_api_key
    api_secret     = var.kafka_api_secret
    bootstrap_url  = var.bootstrap_url
    cluster_id     = var.kafka_cluster_id
  })

  custom_metadata {
    max_versions = 2  # Keep current + previous for dual-credential window
  }
}
```

`max_versions = 2` retains both the current (new) and previous (old) credential versions during the window period.

## Zero-downtime rotation

The dual-credential window relies on Confluent Cloud supporting multiple active API keys per service account. Both old and new keys are valid simultaneously during the 24-hour default window.

<Warning>
  Always rotate consumers before producers. Consumers must be updated to use the new credential and confirmed healthy before the old producer credential is revoked. Revoking a producer credential while consumers still authenticate with it causes authentication failures on message fetch.
</Warning>

<Tabs>
  <Tab title="HashiCorp Vault">

<Steps>
  <Step title="Create a new API key">
    ```bash
    confluent api-key create \
      --resource lkc-xxxxx \
      --service-account sa-xxxxxx \
      --description "Rotated $(date +%Y-%m-%d)"
    ```

    Record the new API key and secret from the output. Both the old and new keys are now active in Confluent Cloud.
  </Step>

  <Step title="Write the new credential to Vault">
    ```bash
    vault kv put secret/fsi-kafka/prod/corebanking/account-txn-producer \
      api_key="NEW_API_KEY" \
      api_secret="NEW_API_SECRET" \
      bootstrap_url="pkc-xxxxx.us-east-1.aws.confluent.cloud:9092" \
      cluster_id="lkc-xxxxx"
    ```

    Vault KV v2 automatically versions the secret. With `max_versions = 2`, both current (new) and previous (old) versions are retained.
  </Step>

  <Step title="Dual-credential window begins">
    Both the old and new API keys are active in Confluent Cloud, and Vault holds both versions. No action is needed — the window has started automatically. The default window duration is 24 hours.
  </Step>

  <Step title="Deploy applications with the new credential">
    Roll out the new credential to all applications consuming from Vault. Choose the approach that matches your deployment:

    ```bash
    # Option A: Rolling restart (Kubernetes)
    kubectl rollout restart deployment/my-kafka-producer -n kafka-apps

    # Option B: Config reload (if supported by application)
    # Application reads new Vault secret on next health check cycle

    # Option C: Vault Agent sidecar (automatic)
    # Vault Agent detects new secret version and renders updated template
    ```
  </Step>

  <Step title="Monitor during the window (24h default)">
    Check for authentication failures during the overlap period:

    ```bash
    confluent audit-log search \
      --start "$(date -u -v-24H +%Y-%m-%dT%H:%M:%SZ)" \
      --resource-type kafka \
      --category AUTHENTICATION \
      | grep -i "failure"
    ```

    Optionally, set a TTL on the previous Vault version to auto-expire after the window:

    ```bash
    vault kv metadata put \
      -custom-metadata="rotation_deadline=$(date -u -v+24H +%Y-%m-%dT%H:%M:%SZ)" \
      secret/fsi-kafka/prod/corebanking/account-txn-producer
    ```
  </Step>

  <Step title="Revoke the old API key">
    After the window expires with no authentication failures:

    ```bash
    # List API keys for the service account
    confluent api-key list --service-account sa-xxxxxx --resource lkc-xxxxx

    # Delete the old API key
    confluent api-key delete OLD_API_KEY_ID
    ```
  </Step>

  <Step title="Verify">
    ```bash
    # Wait 5 minutes, then check audit logs
    sleep 300
    confluent audit-log search \
      --start "$(date -u -v-10M +%Y-%m-%dT%H:%M:%SZ)" \
      --resource-type kafka \
      --category AUTHENTICATION \
      | grep -i "failure"

    # Expected: no failure events
    ```
  </Step>
</Steps>

### Vault TTL configuration

Configure Vault TTL settings to enforce the rotation window automatically:

```bash
# Set default TTL for the secrets engine (90 days)
vault secrets tune -default-lease-ttl=2160h secret/

# Set max TTL with a grace period
vault secrets tune -max-lease-ttl=2400h secret/
```

  <Tab title="Azure Key Vault">

Store Confluent Cloud API keys as Azure Key Vault secrets. Use Managed Identity for access control.

**Rotation approach:**

1. Store the API key and secret as a Key Vault secret (JSON payload).
2. Create an Azure Automation runbook that:
   - Creates a new Confluent API key via the Confluent CLI or API
   - Updates the Key Vault secret with the new key
   - Waits for the dual-credential window (24 hours)
   - Revokes the old API key
3. Schedule the runbook to run every 90 days via Azure Automation.

**Secret format (JSON payload):**

```json
{
  "api_key": "AAAAAAAAAAAAAAAA",
  "api_secret": "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  "bootstrap_url": "pkc-xxxxx.eastus.azure.confluent.cloud:9092",
  "cluster_id": "lkc-xxxxx"
}
```

**Azure CLI example:**

```bash
# Create a new API key
NEW_KEY=$(confluent api-key create \
  --resource lkc-xxxxx \
  --service-account sa-xxxxxx \
  --description "Rotated $(date +%Y-%m-%d)" \
  -o json)

# Write new credential to Key Vault
az keyvault secret set \
  --vault-name fsi-kafka-kv \
  --name "kafka-prod-corebanking-account-txn-producer" \
  --value "$(echo $NEW_KEY | jq -c '{api_key: .key, api_secret: .secret}')"

# After 24h window, revoke the old key
confluent api-key delete OLD_API_KEY_ID
```

<Note>
  Use Azure Key Vault versioning to retain the previous credential version during the window. Azure Key Vault keeps all versions by default — set a retention policy to clean up versions older than the rotation window.
</Note>

  </Tab>

  <Tab title="AWS Secrets Manager">

Store Confluent Cloud API keys as AWS Secrets Manager secrets. Use IAM roles for access control.

**Rotation approach:**

1. Store the API key and secret as a Secrets Manager secret (JSON payload).
2. Create a Lambda rotation function that:
   - Creates a new Confluent API key via the Confluent API
   - Sets the new key as the `AWSPENDING` version stage
   - Tests the new key against the Confluent Cloud cluster
   - Promotes the new key to `AWSCURRENT` and revokes the old key
3. Enable automatic rotation with a 90-day schedule.

**AWS CLI example:**

```bash
# Create a new API key
NEW_KEY=$(confluent api-key create \
  --resource lkc-xxxxx \
  --service-account sa-xxxxxx \
  --description "Rotated $(date +%Y-%m-%d)" \
  -o json)

# Write new credential to Secrets Manager
aws secretsmanager put-secret-value \
  --secret-id "fsi-kafka/prod/corebanking/account-txn-producer" \
  --secret-string "$(echo $NEW_KEY | jq -c '{
    api_key: .key,
    api_secret: .secret,
    bootstrap_url: "pkc-xxxxx.us-east-1.aws.confluent.cloud:9092",
    cluster_id: "lkc-xxxxx"
  }')"

# After 24h window with no failures, revoke the old key
confluent api-key delete OLD_API_KEY_ID
```

**Lambda rotation function stages:**

| Stage | `AWSPENDING` | `AWSCURRENT` | `AWSPREVIOUS` |
|---|---|---|---|
| Before rotation | — | old key | — |
| During window | new key | old key | — |
| After revocation | — | new key | old key |

  </Tab>

  <Tab title="GCP Secret Manager">

Store Confluent Cloud API keys as GCP Secret Manager secrets. Use service account IAM bindings for access control.

**Rotation approach:**

1. Store the API key and secret as a Secret Manager secret version (JSON payload).
2. Create a Cloud Function that:
   - Creates a new Confluent API key via the Confluent API
   - Adds the new key as a new secret version
   - Waits for the dual-credential window (24 hours)
   - Disables the previous secret version and revokes the old API key
3. Trigger the function on a 90-day schedule via Cloud Scheduler.

**gcloud CLI example:**

```bash
# Create a new API key
NEW_KEY=$(confluent api-key create \
  --resource lkc-xxxxx \
  --service-account sa-xxxxxx \
  --description "Rotated $(date +%Y-%m-%d)" \
  -o json)

# Write new credential to Secret Manager
echo "$NEW_KEY" | jq -c '{
  api_key: .key,
  api_secret: .secret,
  bootstrap_url: "pkc-xxxxx.us-central1.gcp.confluent.cloud:9092",
  cluster_id: "lkc-xxxxx"
}' | gcloud secrets versions add \
  "kafka-prod-corebanking-account-txn-producer" \
  --data-file=-

# After 24h window, disable the previous version
PREV_VERSION=$(gcloud secrets versions list \
  "kafka-prod-corebanking-account-txn-producer" \
  --format="value(name)" \
  --filter="state=ENABLED" \
  --sort-by="~createTime" \
  --limit=2 | tail -1)

gcloud secrets versions disable "$PREV_VERSION" \
  --secret="kafka-prod-corebanking-account-txn-producer"

# Revoke the old API key
confluent api-key delete OLD_API_KEY_ID
```

## OAuth token rotation

For applications using OAuth/OAUTHBEARER (the recommended authentication mechanism for Confluent Cloud per ADR-006), token rotation is fully automatic:

- **Token lifetime:** 1 hour (recommended), configured in the identity provider
- **Refresh:** Kafka client libraries automatically refresh tokens before expiration
- **No manual rotation needed:** The IdP issues new tokens and handles key rotation via the JWKS endpoint
- **Revocation:** Disable the user or group in the IdP; all tokens expire naturally within the configured lifetime

OAuth eliminates the operational burden of credential rotation for application workloads. See `scenarios/cc-azure/oauth.tf`, `scenarios/cc-aws/oauth.tf`, and `scenarios/cc-gcp/oauth.tf` for per-scenario IdP configuration.

## Terraform provider credentials

The Confluent Terraform Provider does not support OAuth and requires API keys for authentication (per ADR-006).

- Store provider API keys in CI secrets (GitHub Actions encrypted secrets)
- Rotate quarterly (every 90 days)
- Use a dedicated Cloud API key (not a cluster-scoped key) for the Terraform provider
- After rotation, update the CI secret and verify the next pipeline run succeeds

```bash
# Create a new Cloud API key for the Terraform provider
confluent api-key create --resource cloud

# Update GitHub Actions secrets
gh secret set CONFLUENT_CLOUD_API_KEY --body "NEW_KEY"
gh secret set CONFLUENT_CLOUD_API_SECRET --body "NEW_SECRET"

# Verify: trigger a plan run
gh workflow run terraform-scenario.yml -f scenario-dir=scenarios/cc-azure -f mode=plan
```

## Emergency rotation

If a credential is compromised, skip the dual-credential window and revoke immediately. Accept brief downtime.

<Steps>
  <Step title="Revoke immediately">
    ```bash
    confluent api-key delete COMPROMISED_KEY_ID --force
    ```
  </Step>

  <Step title="Create a new key">
    ```bash
    confluent api-key create \
      --resource lkc-xxxxx \
      --service-account sa-xxxxxx \
      --description "Emergency rotation $(date +%Y-%m-%dT%H:%M:%SZ)"
    ```
  </Step>

  <Step title="Deploy the new key">
    ```bash
    vault kv put secret/fsi-kafka/prod/corebanking/account-txn-producer \
      api_key="NEW_KEY" \
      api_secret="NEW_SECRET" \
      bootstrap_url="..." \
      cluster_id="..."
    ```
  </Step>

  <Step title="Force-restart affected applications">
    ```bash
    kubectl rollout restart deployment \
      -l app.kafka.credential=compromised \
      -n kafka-apps
    ```
  </Step>

  <Step title="Verify no further unauthorized access">
    ```bash
    confluent audit-log search \
      --start "$(date -u -v-1H +%Y-%m-%dT%H:%M:%SZ)" \
      --resource-type kafka \
      --principal "User:COMPROMISED_SA_ID"
    ```
  </Step>

  <Step title="File an incident report">
    Document the compromise, rotation timeline, and blast radius assessment per your organization's incident response procedure.
  </Step>
</Steps>
