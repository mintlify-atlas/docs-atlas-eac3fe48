---
title: "Deploy on Confluent Cloud: AWS, Azure, and GCP"
description: "Terraform deployment of governed Kafka and Schema Registry on Confluent Cloud for AWS, Azure, and GCP with PrivateLink networking."
---

The FSI Data Streaming Platform ships three self-contained Terraform scenarios for Confluent Cloud — `cc-aws`, `cc-azure`, and `cc-gcp` — plus a `private-cloud` scenario for self-managed Confluent Platform clusters reachable via the Confluent REST API. Each scenario uses the same shared `modules/topic` governance module, so topic naming, Avro schema compatibility, RBAC bindings, and SLA-tier defaults are identical across all three clouds. The only differences are in networking (PrivateLink vs Private Endpoint vs PSC), authentication, secrets backend, and Terraform state storage. Azure is the FSI default deployment target because Azure AD OAuth tokens work against both production and DR clusters, eliminating credential swaps during failover.

## Environment variables

All scenarios read configuration from `.env` (copied from `.env.example`). The table below lists the variables used across all cloud providers. Provider-specific variables appear in the cloud tabs that follow.

| Variable | Description |
|---|---|
| `CC_ORG_ID` | Confluent Cloud organization ID (`org-xxxxx`) |
| `CC_ENV_ID` | Confluent Cloud environment ID (`env-xxxxx`) |
| `CC_KAFKA_CLUSTER_ID` | Production Kafka cluster ID (`lkc-xxxxx`) |
| `CC_KAFKA_BOOTSTRAP` | Production Kafka bootstrap endpoint |
| `CC_KAFKA_API_KEY` / `CC_KAFKA_API_SECRET` | Production Kafka credentials (sensitive) |
| `CC_DR_KAFKA_CLUSTER_ID` | DR Kafka cluster ID (`lkc-yyyyy`) |
| `CC_DR_KAFKA_BOOTSTRAP` | DR Kafka bootstrap endpoint |
| `CC_DR_KAFKA_API_KEY` / `CC_DR_KAFKA_API_SECRET` | DR Kafka credentials (sensitive) |
| `CC_SR_CLUSTER_ID` | Schema Registry cluster ID (`lsrc-xxxxx`) |
| `CC_SR_REST_ENDPOINT` | Schema Registry REST endpoint |
| `CC_SR_API_KEY` / `CC_SR_API_SECRET` | Schema Registry credentials (sensitive) |
| `CC_CLUSTER_LINK_NAME` | Cluster Link name for DR mirroring |
| `CLOUD_PROVIDER` | `azure` \| `aws` \| `gcp` |
| `CC_PROD_REGION` | Production cluster region |
| `CC_DR_REGION` | DR cluster region |
| `SECRETS_PROVIDER` | `vault` \| `azure-keyvault` \| `aws-secrets-manager` |
| `TERRAFORM_BACKEND` | `azurerm` \| `s3` \| `gcs` |

## Cloud-specific deployment

<Tabs>
  <Tab title="Azure (default)">

Azure is the FSI default. Azure AD / Entra ID OAuth tokens authenticate against both production and DR clusters over the same credential, so no credential swap is needed during failover.

**Networking:** Azure Private Link creates a Private Endpoint inside your VNet. Configure `AZURE_VNET_ID`, `AZURE_SUBNET_ID`, and `AZURE_PRIVATE_DNS_ZONE` (`privatelink.confluent.cloud`) before applying.

**DR regions:** East US 2 (production) ↔ West US 2 (DR).

**Terraform state:** Azure Blob Storage (`azurerm` backend). Set `TF_BACKEND_RESOURCE_GROUP`, `TF_BACKEND_STORAGE_ACCOUNT`, and `TF_BACKEND_CONTAINER`.

| Concern | Approach | Variable |
|---|---|---|
| Private networking | Azure Private Link → Private Endpoint in VNet | `AZURE_VNET_ID`, `AZURE_SUBNET_ID` |
| Private DNS | Private DNS Zone: `privatelink.confluent.cloud` | `AZURE_PRIVATE_DNS_ZONE` |
| Authentication | Azure AD / Entra ID → OAUTHBEARER | `AZURE_TENANT_ID`, `AZURE_CLIENT_ID` |
| Secrets | Azure Key Vault or HashiCorp Vault | `SECRETS_PROVIDER` |
| Terraform state | Azure Blob Storage (`azurerm` backend) | `TF_BACKEND_STORAGE_ACCOUNT` |
| DR regions | East US 2 ↔ West US 2 | `CC_PROD_REGION`, `CC_DR_REGION` |

<Note>
  **OAuth advantage for DR:** Azure AD tokens work against both clusters. No credential swap is required during failover — a significant operational simplification for FSI DR drills.
</Note>

Create the Blob Storage backend before running `terraform init`:

```bash
# Create resource group
az group create --name rg-terraform-state --location eastus2

# Create storage account
az storage account create \
  --name stterraformstate \
  --resource-group rg-terraform-state \
  --sku Standard_LRS \
  --encryption-services blob

# Create blob container
az storage container create \
  --name tfstate \
  --account-name stterraformstate
```

The `main.tf` `backend "azurerm"` block references these resources:

```hcl
backend "azurerm" {
  resource_group_name  = "rg-terraform-state"   # TF_BACKEND_RESOURCE_GROUP
  storage_account_name = "stterraformstate"      # TF_BACKEND_STORAGE_ACCOUNT
  container_name       = "tfstate"
  key                  = "kafka-platform/prod/terraform.tfstate"
}
```

  </Tab>
  <Tab title="AWS">

**Networking:** AWS PrivateLink exposes Confluent Cloud brokers as VPC Endpoints. AZ ID mapping between your VPC and Confluent Cloud is required — ensure availability zone IDs (not names) align.

**Authentication:** IAM auth or an external identity provider via OAUTHBEARER.

**Secrets:** AWS Secrets Manager. Kafka Connect has a native config provider for Secrets Manager, so connector credentials are never written to disk.

**DR regions:** us-east-1 (production) ↔ us-west-2 (DR).

**Terraform state:** S3 backend with DynamoDB state locking.

| Concern | Approach | Variable |
|---|---|---|
| Private networking | AWS PrivateLink → VPC Endpoint (AZ ID mapping required) | `AWS_VPC_ID`, `AWS_SUBNET_IDS` |
| Authentication | IAM auth or OAUTHBEARER via external IdP | — |
| Secrets | AWS Secrets Manager | `SECRETS_PROVIDER=aws-secrets-manager` |
| Terraform state | S3 backend + DynamoDB lock | `TF_BACKEND_S3_BUCKET`, `TF_BACKEND_DYNAMODB_TABLE` |
| DR regions | us-east-1 ↔ us-west-2 | `CC_PROD_REGION`, `CC_DR_REGION` |

Create the S3 bucket and DynamoDB table before running `terraform init`:

```bash
# Create S3 bucket for state
aws s3api create-bucket \
  --bucket fsi-terraform-state \
  --region us-east-1

# Enable versioning
aws s3api put-bucket-versioning \
  --bucket fsi-terraform-state \
  --versioning-configuration Status=Enabled

# Create DynamoDB table for state locking
aws dynamodb create-table \
  --table-name fsi-terraform-locks \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

The `main.tf` `backend "s3"` block:

```hcl
backend "s3" {
  bucket         = "fsi-terraform-state"       # TF_BACKEND_S3_BUCKET
  key            = "kafka-platform/prod/terraform.tfstate"
  region         = "us-east-1"                 # TF_BACKEND_S3_REGION
  encrypt        = true
  dynamodb_table = "fsi-terraform-locks"       # TF_BACKEND_DYNAMODB_TABLE
}
```

  </Tab>
  <Tab title="GCP">

**Networking:** Private Service Connect (PSC) exposes Confluent Cloud brokers via a single IP address. DNS must be configured manually to resolve broker hostnames to that IP.

**Authentication:** Google Cloud Identity or Workload Identity Federation.

**Secrets:** Google Secret Manager.

**DR regions:** us-east1 (production) ↔ us-west1 (DR).

**Terraform state:** GCS backend. GCS provides built-in state locking — no additional infrastructure required.

<Note>
  GCP uses region names without digit hyphens: `us-east1` not `us-east-1`. Set `CC_PROD_REGION=us-east1` and `CC_DR_REGION=us-west1`.
</Note>

| Concern | Approach | Variable |
|---|---|---|
| Private networking | Private Service Connect (PSC), single IP, manual DNS | `GCP_NETWORK`, `GCP_SUBNETWORK` |
| Authentication | Google Cloud Identity / Workload Identity Federation | — |
| Secrets | Google Secret Manager | `SECRETS_PROVIDER` |
| Terraform state | GCS backend (built-in locking) | `TF_BACKEND_GCS_BUCKET`, `TF_BACKEND_GCS_PREFIX` |
| DR regions | us-east1 ↔ us-west1 | `CC_PROD_REGION`, `CC_DR_REGION` |

Create the GCS bucket before running `terraform init`:

```bash
# Create GCS bucket for state
gsutil mb -l us-east1 gs://fsi-terraform-state

# Enable versioning
gsutil versioning set on gs://fsi-terraform-state
```

The `main.tf` `backend "gcs"` block:

```hcl
backend "gcs" {
  bucket = "fsi-terraform-state"               # TF_BACKEND_GCS_BUCKET
  prefix = "kafka-platform/prod"               # TF_BACKEND_GCS_PREFIX
}
```

  </Tab>
</Tabs>

## Deployment steps

<Steps>
  <Step title="Clone and configure">

    Change into the scenario directory for your cloud provider and copy the configuration templates:

    ```bash
    cd scenarios/cc-azure/   # or cc-aws, cc-gcp

    cp clusters.auto.tfvars.example clusters.auto.tfvars
    cp terraform.tfvars.example terraform.tfvars
    ```

    Edit `clusters.auto.tfvars` with your Confluent Cloud cluster IDs, REST endpoints, and CRNs. Edit `terraform.tfvars` with your API keys and secrets. The `clusters.auto.tfvars` file is auto-loaded by Terraform due to the `*.auto.tfvars` naming pattern.

    Example `clusters.auto.tfvars` (Azure):

    ```hcl
    kafka_cluster_id       = "lkc-xxxxx"
    kafka_rest_endpoint    = "https://pkc-xxxxx.eastus2.azure.confluent.cloud:443"
    kafka_cluster_crn      = "crn://confluent.cloud/organization=org-xxxxx/environment=env-xxxxx/cloud-cluster=lkc-xxxxx"
    sr_cluster_id          = "lsrc-xxxxx"
    sr_rest_endpoint       = "https://psrc-xxxxx.eastus2.azure.confluent.cloud"
    sr_cluster_crn         = "crn://confluent.cloud/organization=org-xxxxx/environment=env-xxxxx/schema-registry=lsrc-xxxxx"
    cluster_link_name      = "cluster-link-bidir-prod-dr"
    dr_kafka_cluster_id    = "lkc-yyyyy"
    dr_kafka_rest_endpoint = "https://pkc-yyyyy.westus2.azure.confluent.cloud:443"
    ```

  </Step>
  <Step title="Create backend infrastructure">

    Create the cloud-specific backend resources (S3 + DynamoDB, Azure Blob Storage, or GCS bucket) using the commands in the cloud tab above. Update the `backend` block in `main.tf` with your resource names.

  </Step>
  <Step title="Initialize Terraform">

    ```bash
    terraform init
    ```

    Terraform downloads the `confluentinc/confluent` provider (`~> 2.0`) and configures the remote backend. The provider version is pinned in `main.tf`.

  </Step>
  <Step title="Review the plan">

    ```bash
    terraform plan
    ```

    Review the planned resources. Each topic module creates: a Kafka topic, an Avro schema in Schema Registry, RBAC role bindings for producer and consumer service accounts, and a DR mirror topic via Cluster Linking.

  </Step>
  <Step title="Apply">

    ```bash
    terraform apply
    ```

    Confirm when prompted. CI/CD applies automatically on merge to `main` via `.github/workflows/terraform-apply.yml`.

  </Step>
  <Step title="Verify">

    ```bash
    # List all created resources
    terraform output -json | jq 'keys'

    # Inspect a specific topic
    terraform output -json corebanking_account_txn
    ```

  </Step>
</Steps>

## Topic declarations

Topic governance is declared using the shared `modules/topic` module. Each module call produces a Kafka topic, Avro schema, RBAC bindings, and a DR mirror. The `sla_tier` input drives partition count, retention, and schema compatibility automatically.

```hcl
module "corebanking_account_txn" {
  source         = "../../modules/topic"
  domain         = "corebanking"
  application    = "transactions"
  schema_version = "v1"
  entity         = "account-transaction"
  owner          = "core-banking-team@company.com"
  sla_tier       = "critical"
  schema_file    = "../../schemas/examples/account-transaction.avsc"
  pii_fields     = ["member_name", "ssn_last4", "account_number"]
  producer_service_accounts = ["sa-corebanking-producer"]
  consumer_service_accounts = ["sa-corebanking-consumer", "sa-fraud-enrichment"]

  kafka_cluster_id               = local.infra.kafka_cluster_id
  kafka_rest_endpoint            = local.infra.kafka_rest_endpoint
  kafka_api_key                  = var.kafka_api_key
  kafka_api_secret               = var.kafka_api_secret
  kafka_cluster_crn              = local.infra.kafka_cluster_crn
  schema_registry_cluster_id     = local.infra.sr_cluster_id
  schema_registry_rest_endpoint  = local.infra.sr_rest_endpoint
  schema_registry_api_key        = var.sr_api_key
  schema_registry_api_secret     = var.sr_api_secret
  schema_registry_cluster_crn    = local.infra.sr_cluster_crn
  cluster_link_name              = local.infra.cluster_link_name
  dr_kafka_cluster_id            = local.infra.dr_kafka_cluster_id
  dr_kafka_rest_endpoint         = local.infra.dr_kafka_rest_endpoint
  dr_kafka_api_key               = var.dr_kafka_api_key
  dr_kafka_api_secret            = var.dr_kafka_api_secret
}
```

Reference topic declarations for `corebanking`, `fraud`, and `compliance` domains are provided in `example-topics.tf` in each scenario directory.

## Private Cloud (self-managed CP)

The `scenarios/private-cloud/` scenario applies the same governance module to a pre-existing self-managed Confluent Platform cluster. The Confluent Terraform provider connects via `kafka_rest_endpoint` instead of `cloud_api_key`, targeting the Kafka REST Proxy (MDS or standalone) directly.

```hcl
provider "confluent" {
  kafka_rest_endpoint = var.cp_kafka_rest_endpoint
  kafka_api_key       = var.cp_kafka_api_key
  kafka_api_secret    = var.cp_kafka_api_secret

  schema_registry_rest_endpoint = var.cp_sr_rest_endpoint
  schema_registry_api_key       = var.cp_sr_api_key
  schema_registry_api_secret    = var.cp_sr_api_secret
}
```

Configure endpoints in `terraform.tfvars`:

```hcl
cp_kafka_rest_endpoint = "https://kafka-east-1.example.com:8090"
cp_kafka_api_key       = "<your-kafka-rest-api-key>"
cp_kafka_api_secret    = "<your-kafka-rest-api-secret>"
cp_sr_rest_endpoint    = "https://sr-east-1.example.com:8081"
cp_sr_api_key          = "<your-sr-api-key>"
cp_sr_api_secret       = "<your-sr-api-secret>"
cp_kafka_cluster_id    = "kafka-east-cluster"
```

Private Cloud clusters use MDS or LDAP-backed REST credentials. RBAC is managed via MDS role bindings using the same Terraform provider resources as Confluent Cloud. CRN fields are left empty since private CP clusters do not use CRNs.
