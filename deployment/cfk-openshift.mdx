---
title: "Deploy Confluent for Kubernetes on OpenShift"
description: "Helm-based deployment of Confluent for Kubernetes on OpenShift 4.x, with mTLS, cert-manager, KafkaTopic CRDs, MirrorMaker 2, and Flink Kubernetes Operator."
---

The `scenarios/cfk-openshift/` scenario deploys a complete Kafka platform on OpenShift Container Platform (OCP) 4.14–4.21 using the Confluent for Kubernetes (CFK) operator. The scenario is self-contained: Kafka brokers, Schema Registry, Kafka Connect, a dedicated MirrorMaker 2 cluster for DR replication, and the Flink Kubernetes Operator 1.14.0 are all declared as Helm values and CRDs. Authentication uses mutual TLS managed by CFK with cert-manager handling certificate lifecycle. Topic governance enforces the same FSI naming conventions and SLA-tier defaults as the Terraform-based Confluent Cloud scenarios — the only format difference is `KafkaTopic` CRDs instead of Terraform module calls.

## Prerequisites

- OpenShift 4.14–4.21 with `cluster-admin` access
- Helm 3.x and `oc` or `kubectl` CLI
- cert-manager 1.x installed cluster-wide (most OCP clusters include this)
- Docker daemon accessible for building the custom Flink image

## Install the CFK operator

Two installation paths are available:

<Tabs>
  <Tab title="Helm (recommended for production)">

    Helm provides full control over operator configuration and is recommended for production environments and air-gapped clusters.

    ```bash
    helm repo add confluentinc https://packages.confluent.io/helm
    helm repo update
    helm upgrade --install confluent-operator \
      confluentinc/confluent-for-kubernetes \
      --namespace confluent --create-namespace
    ```

    Verify the operator pod is Ready before proceeding:

    ```bash
    oc get pods -n confluent -w
    ```

  </Tab>
  <Tab title="OLM (OperatorHub)">

    Install from the OpenShift OperatorHub UI when air-gapped access or full Helm customization is not required:

    1. Navigate to **Operators > OperatorHub** in the OpenShift console.
    2. Search for **Confluent for Kubernetes**.
    3. Click **Install**, select the `confluent` namespace.
    4. Wait for the operator pod to become Ready.

    <Note>
      Configuration options are limited when installed via OperatorHub. For production deployments requiring custom operator configuration, use the Helm path.
    </Note>

  </Tab>
</Tabs>

## Deploy Kafka, Schema Registry, and Connect

<Steps>
  <Step title="Apply component CRs">

    Apply the Kafka, Schema Registry, Connect, and MirrorMaker 2 custom resources in order:

    ```bash
    oc apply -f values/kafka-rest-class.yaml
    oc apply -f values/kafka.yaml
    oc apply -f values/schemaregistry.yaml
    oc apply -f values/connect.yaml
    oc apply -f values/connect-mm2.yaml
    ```

    Wait for all pods to become Ready:

    ```bash
    oc get pods -n confluent -w
    ```

  </Step>
  <Step title="Apply KafkaTopic CRDs">

    Topic governance is declared as `KafkaTopic` CRDs. Apply all topic definitions:

    ```bash
    oc apply -f topics/
    ```

    <Warning>
      All topic changes must go through `KafkaTopic` CRDs. The CFK operator continuously reconciles CRDs against actual Kafka state. Manual topic changes made via CLI or Control Center will be **reverted** by the reconciliation loop.
    </Warning>

  </Step>
  <Step title="Apply ACLs">

    Review and adapt ACL templates for your service account principals, then apply:

    ```bash
    oc apply -f acls/acl-templates.yaml
    ```

    For Confluent RBAC via MDS, use the alternative configuration:

    ```bash
    oc apply -f acls/acl-mds-alternative.yaml
    ```

  </Step>
  <Step title="Verify deployment">

    Confirm all resources are running:

    ```bash
    oc get kafka,schemaregistry,connect -n confluent
    oc get kafkatopic -n confluent
    ```

  </Step>
</Steps>

## Kafka CR configuration

The `values/kafka.yaml` CR deploys a 3-broker cluster with mTLS authentication, ACL authorization, and JMX metrics for Prometheus.

```yaml
apiVersion: platform.confluent.io/v1beta1
kind: Kafka
metadata:
  name: kafka
  namespace: confluent
spec:
  replicas: 3
  image:
    application: confluentinc/cp-server:7.6.0
  dataVolumeCapacity: 100Gi
  listeners:
    internal:
      authentication:
        type: mtls
        principalMappingRules:
          - "RULE:.*CN[\\s]?=[\\s]?([a-zA-Z0-9.]*)?.*/$1/"
      tls:
        enabled: true
    external:
      authentication:
        type: mtls
      tls:
        enabled: true
  authorization:
    type: simple
    superUsers:
      - User:kafka
      - User:connect
      - User:schemaregistry
  tls:
    autoGeneratedCerts: true
  configOverrides:
    server:
      - auto.create.topics.enable=false
      - default.replication.factor=3
      - min.insync.replicas=2
```

## KafkaTopic CRD format

KafkaTopic CRDs carry the same FSI governance labels as Terraform topic modules and CPTopic YAML files. The `fsi.sla-tier` label drives partition and retention defaults:

```yaml
apiVersion: platform.confluent.io/v1beta1
kind: KafkaTopic
metadata:
  name: corebanking.transactions.v1.account-transaction
  namespace: confluent
  labels:
    fsi.sla-tier: critical
    fsi.domain: corebanking
    fsi.owner: corebanking-team@fsi.org
    fsi.data-classification: confidential
spec:
  replicas: 3
  partitionCount: 12
  kafkaClusterRef:
    name: kafka
  configs:
    cleanup.policy: "delete"
    retention.ms: "604800000"
```

Reference topics for `corebanking`, `fraud`, and `compliance` domains are provided in `topics/`.

## mTLS and cert-manager

mTLS is the authentication mechanism for all inter-component and client-to-broker communication. CFK integrates with cert-manager to automate certificate generation and rotation:

- `tls.autoGeneratedCerts: true` instructs CFK to request certificates from cert-manager automatically.
- cert-manager handles the full TLS certificate lifecycle (issuance, renewal, distribution).
- Principal mapping extracts the CN from the client certificate as the Kafka identity, which is then matched against ACL rules.

All inter-component connections (Schema Registry to Kafka, Connect to Kafka) use the internal mTLS listener.

<Note>
  cert-manager must be installed cluster-wide before deploying CFK components with `autoGeneratedCerts: true`. Most OCP clusters include cert-manager. Verify with `oc get pods -n cert-manager`.
</Note>

## Authorization modes

<Tabs>
  <Tab title="Kafka ACLs (default)">

    The default authorization mode is `simple` (Kafka ACLs). No external infrastructure is required. Producer and consumer principals are extracted from mTLS certificate CNs and matched against ACL rules in `acls/acl-templates.yaml`.

    This maps to the same `DeveloperWrite` / `DeveloperRead` patterns used in Confluent Cloud RBAC bindings.

  </Tab>
  <Tab title="MDS RBAC (advanced)">

    For teams requiring CC-equivalent role bindings with centralized identity management, Confluent RBAC via the Metadata Service (MDS) provides full role-based access control. MDS requires an LDAP/AD backend for identity resolution.

    See `acls/acl-mds-alternative.yaml` for MDS configuration reference.

  </Tab>
</Tabs>

## Ansible roles for CFK

Two Ansible roles in the platform's `ansible/roles/` directory apply to CFK deployments:

| Role | Purpose |
|---|---|
| `cfk_operator` | CFK Helm deployment with CR readiness gates — waits for all CRDs to become Ready before proceeding |
| `cfk_topic` | Generates `KafkaTopic` CRDs from CPTopic YAML format, preserving SLA-tier governance labels |

Both roles support `--check` mode for audit-ready dry runs:

```bash
# Deploy CFK via Ansible (includes readiness gating)
ansible-playbook site.yml --tags cfk

# Dry-run: preview what would change
ansible-playbook site.yml --tags cfk --check
```

## Flink Kubernetes Operator

The Flink Kubernetes Operator 1.14.0 is bundled in this scenario for stream processing parity with Confluent Cloud Flink. A single scenario provides the complete platform: Kafka + Schema Registry + Connect + MirrorMaker 2 + Flink.

<Steps>
  <Step title="Install the Flink Kubernetes Operator">

    ```bash
    helm repo add flink-operator \
      https://archive.apache.org/dist/flink/flink-kubernetes-operator-1.14.0/
    helm install flink-kubernetes-operator \
      flink-operator/flink-kubernetes-operator \
      --namespace flink --create-namespace \
      -f flink/flink-operator-values.yaml
    ```

  </Step>
  <Step title="Build the custom Flink image">

    The base Flink image does not include the `avro-confluent` format connector JARs needed for Schema Registry integration. Build the custom image before deploying FlinkDeployment CRDs:

    ```bash
    docker build -t fsi-flink:1.20 flink/flink-docker/
    ```

    <Tip>
      For air-gapped environments, download the connector JARs locally and modify the Dockerfile to use `COPY` instead of `wget`. See comments in `flink/flink-docker/Dockerfile`.
    </Tip>

  </Step>
  <Step title="Deploy FlinkDeployment CRDs">

    ```bash
    kubectl apply -f flink/flink-session-cluster.yaml -n flink
    kubectl apply -f flink/examples/tumbling-window.yaml -n flink
    kubectl apply -f flink/examples/stream-table-join.yaml -n flink
    kubectl apply -f flink/examples/filter-and-route.yaml -n flink
    ```

  </Step>
</Steps>

Unlike Confluent Cloud Flink (which auto-discovers Schema Registry subjects), CFK Flink requires explicit `CREATE TABLE` statements with the `avro-confluent` format connector:

```sql
CREATE TABLE account_transactions (
  transaction_id STRING,
  account_number STRING,
  amount DOUBLE,
  event_time TIMESTAMP(3)
) WITH (
  'connector' = 'kafka',
  'topic' = 'corebanking.transactions.v1.account-transaction',
  'properties.bootstrap.servers' = 'kafka.confluent.svc.cluster.local:9092',
  'format' = 'avro-confluent',
  'avro-confluent.url' = 'https://schemaregistry.confluent.svc.cluster.local:8081'
);
```

## Observability

CFK exposes Kafka JMX metrics on port 7778 via the built-in Prometheus JMX exporter (no sidecar required). Apply the PodMonitor to register Kafka as a Prometheus scrape target:

```bash
oc apply -f observability/pod-monitor-kafka.yaml
oc apply -f observability/pod-monitor-flink.yaml
```

Flink uses the built-in `PrometheusReporterFactory` on port 9249. Import dashboard templates from `observability/{provider}/` for Grafana, Dynatrace, Datadog, Splunk, New Relic, or IBM Instana.

## What gets created

| Resource | Kind | Replicas | Purpose |
|---|---|---|---|
| `kafka` | Kafka | 3 | Brokers with mTLS auth and ACL authorization |
| `schemaregistry` | SchemaRegistry | 2 | Schema Registry with TLS |
| `connect` | Connect | 2 | Application connectors (JDBC, etc.) |
| `connect-mm2` | Connect | 2 | Dedicated MirrorMaker 2 cluster for DR |
| `corebanking.transactions.v1.account-transaction` | KafkaTopic | — | Critical tier reference topic |
| `fraud.alerts.v1.signal` | KafkaTopic | — | Critical tier reference topic |
| `compliance.screening.v1.result` | KafkaTopic | — | Compliance tier reference topic |
| `mm2-source-east-west` | Connector | — | MirrorSourceConnector for DR replication |
| `mm2-checkpoint-east-west` | Connector | — | MirrorCheckpointConnector for offset sync |

## OpenShift-specific notes

<AccordionGroup>
  <Accordion title="SecurityContextConstraints (SCC)">
    CFK 3.2 supports OCP 4.14–4.21. If pods fail to start with permission errors, verify that SecurityContextConstraints allow the security context required by CFK pods. CFK's default SCC requirements are documented in the Confluent for Kubernetes release notes.
  </Accordion>
  <Accordion title="External access">
    Configure OpenShift Routes or LoadBalancer services for external Kafka access. Internal cluster communication uses the CFK-managed service DNS names (e.g., `kafka.confluent.svc.cluster.local`).
  </Accordion>
  <Accordion title="MirrorMaker 2 DR topology">
    Configure the source cluster bootstrap endpoint in the MM2 connector CRDs (`mm2/mm2-source-connector.yaml`) for cross-cluster replication. The DR region cluster endpoint must be reachable from the OpenShift cluster network.
  </Accordion>
</AccordionGroup>
